## Learned Patterns

### High Confidence Unsubscribe
- `newsletter@` or `marketing@` sender prefix (+4 alone, crosses threshold)
- `Precedence: bulk` + body contains "unsubscribe" = marketing (score 4)
- Proton `no-reply@notify.proton.me` / `no-reply@offers.proton.me` = upsell marketing
- `news@` prefix on commercial domains
- Black Friday / sale keywords in subject + bulk precedence

### High Confidence Keep
- Personal correspondence (In-Reply-To header)
- Receipts, invoices, order confirmations
- Government emails (myGov)
- Banking account setup (St.George)
- Legal/conveyancing (kjblaw.com.au)
- Payslips (Xero)
- Docusign signing requests

### High Confidence Notification
- Google Cloud Alerting (100% confidence)
- Firebase crash reports (75%)
- Google Calendar notifications (88%)
- Mail Delivery Subsystem bounces (50%)
- Slack channel updates (88%)
- GitHub app authorizations (100%)
- ActewAGL utility bills (63%)

### Unsubscribe Method Notes
- No List-Unsubscribe headers found on most Proton emails (Bridge may not expose them or senders don't include them)
- Unsubscribe attempts rely on body links (not automated via CLI)
- Most marketing trashed without unsubscribe due to missing headers

### Session 2 Fixes
- Split `isImportant()` patterns: sender domain patterns check `from` only, keyword patterns check `subject` only
- Prevents false positives: "transactional" in domain (`@transactional.getlate.dev`) no longer triggers, "Delivery" in marketing subject no longer triggers
- Added `hello@`, `memberships?@`, `travel-insider@`, `@*edm*` as weak marketing sender patterns (+2)
- Added `firebaseapptesters.com` and `youtube.com` to notification detector
- Scanner now prioritizes notification over marketing when both detectors fire (prevents trashing GitHub/Firebase/YouTube)
- Fixed Apple domain regex: `$` anchor → `\b` to handle `"Display Name" <user@domain>` format
- Word "Sale" in legal context (kjblaw.com.au conveyancing) = false positive -- manually excluded

### Proton-Specific Notes
- Proton Bridge injects `References: <...@protonmail.internalid>` on all messages -- must be excluded from conversation thread detection
- Bridge uses STARTTLS (not implicit TLS) on ports 1143/1025
- No CATEGORY_PROMOTIONS equivalent in Proton -- rely on Precedence + body unsubscribe + sender patterns
- Folders/Notifications path (not just "Notifications")
